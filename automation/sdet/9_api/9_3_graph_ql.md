# 📘 Глава 9. Архитектура API и Микросервисы
## Тема 9.3. GraphQL в микросервисах: Специфика тестирования мутаций и запросов

### Предисловие:
В 2025–2026 годах микросервисная архитектура массово использует **GraphQL** в качестве паттерна BFF (Backend-for-Frontend) или API Gateway. Это позволяет клиентским приложениям (Web, Mobile) забирать данные из десятков микросервисов за один сетевой запрос, избегая проблем Over-fetching (избыточной выборки) и Under-fetching (недостаточной выборки) [1][5].

Однако автоматизация тестирования GraphQL в корне отличается от REST. Даже международные сертификационные центры (AT*SQA, связанные с ISTQB) выделили GraphQL в отдельную сертификацию по API-тестированию, соответствующую стандартам управления качеством ISO 9001:2015 [6].

### Введение: Главное архитектурное отличие GraphQL от REST
В REST вы обращаетесь к разным URL (`/users`, `/orders`) с помощью разных HTTP-методов (GET, POST, PUT, DELETE). 
В GraphQL существует **только один Endpoint** (обычно `/graphql`), и 99% запросов отправляются через метод **POST** [3][4]. 

**Критическая особенность для SDET:** GraphQL *почти всегда* возвращает HTTP-статус `200 OK`, даже если внутри произошла ошибка бизнес-логики или сбой микросервиса [4]. Поэтому классические проверки REST-тестов вида `assert response.status_code == 200` в GraphQL абсолютно бесполезны.

### Часть 1. Мутации и Запросы: Что именно мы тестируем?

GraphQL-схема (Schema) строго типизирована и делит все операции на две категории:

#### 1. Queries (Запросы - Чтение данных)
Аналог `GET` в REST. Используются для выборки данных без изменения состояния на сервере.
*   **Специфика тестирования:** Клиент сам решает, какие поля ему нужны. SDET должен тестировать не только "полный" запрос, но и динамические комбинации. Если тест запрашивает только поле `id`, бэкенд (Resolver) не должен делать лишний JOIN к таблице профилей в базе данных (оптимизация производительности) [2][5].

#### 2. Mutations (Мутации - Изменение данных)
Аналог `POST`, `PUT`, `DELETE` в REST. Используются для создания, обновления или удаления данных.
*   **Специфика тестирования:** Мутация не только изменяет данные, но и *сразу же возвращает обновленный объект* [3]. SDET должен писать тесты так, чтобы в одном HTTP-запросе передать параметры (Variables) для изменения данных и сразу запросить вложенные поля для Assert'а (например, обновить имя пользователя и сразу запросить его новый профиль).

### Часть 2. Стратегия тестирования GraphQL (Уровни проверок)

В 2026 году тестирование GraphQL разбивается на три строгих уровня [2][3]:

1.  **Schema Validation (Валидация контракта):**
    Поскольку GraphQL строго типизирован, схема — это единственный источник истины (Single Source of Truth). Senior SDET внедряет в CI/CD инструменты вроде `graphql-schema-linter` или Apollo Studio [2][3]. Если разработчик удалил обязательное поле из схемы, пайплайн должен упасть *до* прогона E2E-тестов[2].
2.  **Resolver Testing (Тестирование резолверов):**
    Резолверы — это backend-функции, которые физически "добывают" данные из БД или других микросервисов. На этом уровне применяются инструменты вроде EasyGraphQL Tester для написания изолированных интеграционных тестов без полного запуска сервера [3].
3.  **End-to-End Dynamic Query Testing:**
    Отправка реальных запросов и парсинг JSON-ответов. SDET всегда проверяет наличие массива `"errors"` в теле ответа.
    ```python
    # Пример правильного GraphQL Assert'а на Python (2026)
    def test_create_user(graphql_client):
        mutation = """
        mutation CreateUser($input: UserInput!) {
            createUser(input: $input) {
                id
                email
            }
        }
        """
        variables = {"input": {"email": "test@mail.com"}}
        response = graphql_client.post("/graphql", json={"query": mutation, "variables": variables})
        
        # 1. Сначала проверяем, что нет скрытых ошибок в теле
        body = response.json()
        assert "errors" not in body, f"GraphQL Error: {body.get('errors')}"
        
        # 2. Затем проверяем сами данные
        assert body["data"]["createUser"]["email"] == "test@mail.com"
    ```

### Часть 3. Edge Cases: Безопасность и Производительность (ISO/IEC 25010)

Главные уязвимости GraphQL возникают из-за его гибкости. На собеседованиях FAANG SDET обязан знать, как тестировать API на устойчивость к отказам (Security & Performance)[2].

#### 1. Проблема N+1 (The N+1 Problem)
*   **Сценарий:** Вы запрашиваете список из 100 пользователей, и для каждого просите вернуть его последние заказы. Неоптимизированный резолвер сделает 1 запрос к таблице пользователей и 100 отдельных запросов к таблице заказов (Итого: 101 запрос к БД). Это убьет базу данных (Performance Bottleneck) [1].
*   **Как тестировать:** SDET должен использовать профайлеры или метрики (например, Apollo Tracing) во время автотестов, чтобы гарантировать, что разработчики внедрили паттерн **DataLoader** (который батчит 100 запросов в 1 запрос с `IN (...)` оператором) [1].

#### 2. Deep Nesting & Query Complexity (Атака вложенностью)
*   **Сценарий:** Хакер (или некорректный клиент) отправляет циклический запрос бесконечной глубины: `User -> Friends -> User -> Friends...`. Это вызовет Denial of Service (DoS) и падение сервера по OOM (Out of Memory)[1][2].
*   **Как тестировать:** Автотест намеренно генерирует сверх-глубокий запрос. Ожидаемый результат: сервер должен отклонить запрос с ошибкой `"Max query depth exceeded"`. Настройка **Query Complexity Limits** — обязательный стандарт безопасности 2026 года [2][4].

#### 3. Introspection Attacks (Утечка схемы)
*   **Сценарий:** GraphQL имеет фичу "Интроспекция" (Introspection), которая позволяет клиенту запросить у сервера полное описание всех доступных методов и таблиц.
*   **Как тестировать:** В тестовом стенде это полезно, но SDET обязан написать Security-тест, проверяющий, что отправка универсального запроса `query { __schema { types { name } } }` на **Production-окружение** возвращает ошибку доступа [2].

### Часть 4. Best Practices & Anti-patterns (SDET Контекст)

#### ❌ Anti-patterns (Антипаттерны разработки и тестирования)
1.  **ORMQL (GraphQL как ORM):** Проброс GraphQL-запросов напрямую в базу данных без слоя бизнес-логики. Это приводит к тесной связности (Tight Coupling) и невозможности безопасно мокать данные в автотестах.
2.  **Over-fetching в тестах:** Если тест должен проверить только создание `ID`, но в мутации запрашивается 50 тяжелых полей (фотографии, история, адреса) "на всякий случай". Это замедляет прогон тестов (Inefficient querying) [1].
3.  **Хардкод запросов в строках:** Сборка GraphQL-запросов через f-строки Python (`f"query {{ user(id: {user_id}) }}"`). Это нарушает синтаксис при сложных типах. **Всегда используйте `Variables` (переменные)** и передавайте их как отдельный JSON-объект в payload [4].

#### ✅ Best Practices (Стандарты 2026)
1.  **Тестирование "Негативных параметров":** Обязательно проверяйте, как API ведет себя при передаче в мутацию типов, которые не поддерживаются схемой. Грамотный GraphQL-сервер должен вернуть `400 Bad Request` на этапе предварительной валидации (Parsing & Validation phase), даже не доходя до резолверов [4].
2.  **Использование библиотек для генерации запросов:** Senior SDET интегрирует инструменты вроде `sgqlc` или Python-клиенты `gql`, которые автоматически генерируют классы Python на основе скачанной схемы. Это позволяет писать автотесты со строгой типизацией (Type Hinting) и автодополнением в IDE.
3.  **Мокирование зависимостей (Keploy / EasyGraphQL):** В микросервисной архитектуре GraphQL-сервер часто является лишь "фасадом". Для тестирования логики самого GraphQL-шлюза (Gateway) все нижестоящие REST/gRPC микросервисы должны быть замоканы с использованием современных инструментов (например, Keploy), чтобы сфокусироваться исключительно на проверке Мутаций и Запросов[3][5].
