# 📘 Глава 1. Core Python (Уровень Архитектора)
## Тема 1.2. Многопоточность и Многопроцессность: GIL, threading vs multiprocessing, Asyncio

### Введение: Контекст международных стандартов
Согласно стандарту **ISO/IEC 25010:2023**, характеристика **Performance Efficiency** включает в себя **Time Behavior** (Время отклика) и **Capacity** (Пропускная способность). В контексте Test Automation (ISTQB Advanced Level TAE) это означает, что архитектура фреймворка должна минимизировать время прогона (Execution Time) за счет правильного распараллеливания. Если ваши 10 000 API-тестов идут 2 часа вместо 2 минут — проблема не в сервере, а в неправильно выбранной модели конкурентности.

В Python существует три парадигмы выполнения задач: **Threading**, **Multiprocessing** и **Asyncio**. Чтобы выбрать правильную, нужно понимать, что такое GIL и как он изменился в 2026 году.

---

### Часть 1. GIL (Global Interpreter Lock): Конец эпохи

**Что такое GIL исторически?**
GIL (Глобальная блокировка интерпретатора) — это мьютекс, который защищал внутренние структуры памяти CPython (тот самый Reference Count из Темы 1.1) от состояния гонки (Race Condition). Из-за GIL **только один поток (thread) мог выполнять байт-код Python в любой момент времени**, даже если у вас 64-ядерный процессор. 
*Итог до 2025 года:* Многопоточность в Python не ускоряла задачи, требующие вычислений процессора (CPU-bound).

🔥 **Реалии 2026 года: Free-Threaded Python (PEP 703 & PEP 779)**
В релизе **Python 3.14** (и развивающемся 3.15) сборки без GIL (Free-threaded Python) официально перешли из статуса "экспериментальных" в статус **"Официально поддерживаемых" (Phase II)**. 
*   Теперь вы можете запустить тесты командой `python3.14t` (где `t` — free-threaded), и ваши потоки (`threading`) будут выполняться **истинно параллельно** на разных ядрах CPU.
*   Single-thread деградация производительности снизилась до ~5-10%, а прирост на многоядерных задачах достигает 300%+.

*Важно для Senior SDET:* Несмотря на поддержку No-GIL, дефолтная сборка Python пока поставляется с GIL (ради обратной совместимости старых C-библиотек). Поэтому инженер должен уметь работать в обеих парадигмах.

---

### Часть 2. Threading (Многопоточность)

Модуль `threading` запускает задачи в одном процессе ОС, разделяя общую память.

**Когда использовать (GIL включен):**
*   **I/O-bound задачи:** Когда тест большую часть времени "ждет". Например, ожидание ответа от базы данных, скачивание файлов, селен-грид (Selenium Grid) запросы. Пока один поток ждет ответа сети, GIL отпускается, и другой поток начинает работу.

**Когда использовать (Free-threaded Python 3.14t - GIL отключен):**
*   **CPU-bound задачи:** Парсинг гигантских JSON-ответов, обработка ML-датасетов, криптографическое хеширование тестовых токенов. Теперь потоки могут делать это параллельно.

**Специфика и риски:**
Без GIL возникает реальная угроза **Race Conditions** на уровне бизнес-логики. Если два потока автотестов одновременно модифицируют один глобальный словарь `test_results`, данные будут повреждены. Необходимо использовать примитивы синхронизации: `threading.Lock()` или `threading.Semaphore()`.

---

### Часть 3. Multiprocessing (Многопроцессность)

Модуль `multiprocessing` обходит классический GIL путем создания совершенно **новых независимых процессов ОС**. Каждый процесс получает свой собственный интерпретатор Python, свой GIL и свою изолированную память.

**Когда использовать в 2026 году:**
1.  **Полная изоляция данных:** Когда вы запускаете UI-тесты с использованием тяжелых браузеров и хотите гарантировать, что крах (Segmentation Fault) одной вкладки или утечка памяти в одном тесте не убьет весь тестовый прогон. (На этом построен плагин `pytest-xdist`).
2.  **Legacy C-extensions:** Если ваш фреймворк использует старые внутренние библиотеки (или старые версии NumPy/Pandas), которые еще не адаптированы под Free-threaded Python (отсутствует ABI тег `abi3t`).

**Главный недостаток:**
Очень "дорого" по ресурсам (High Overhead). Создание процесса занимает время, а потребление RAM возрастает кратно (10 процессов = 10 копий интерпретатора в ОЗУ). Обмен данными между тестами (IPC - Inter-Process Communication) сложен и требует использования `Queue` или `Pipes`, что медленно из-за сериализации (Pickling).

### Часть 4. Asyncio (Асинхронность) — Стандарт современного SDET

`asyncio` не создает ни новых процессов, ни новых потоков ОС. Все работает в **одном потоке**, используя Event Loop (Цикл событий) и кооперативную многозадачность.

**Принцип работы:** 
Когда автотест отправляет API-запрос (`await client.get(...)`), он добровольно говорит Event Loop'у: *"Я буду ждать ответа сети 2 секунды. Забери у меня управление и отдай другому тесту"*.

**Почему это стандарт 2026 года:**
Инструменты вроде **Playwright** и **httpx** изначально построены вокруг `asyncio`. Эта парадигма позволяет запустить 10 000 API-запросов одновременно, потребляя всего ~50 МБ оперативной памяти (в отличие от `multiprocessing`, который на 10 000 процессах просто уничтожит сервер).

### Часть 5. Сравнительная матрица (Cheatsheet Архитектора)

| Характеристика | `threading` (Classic) | `threading` (Free-threaded 3.14t) | `multiprocessing` | `asyncio` |
| :--- | :--- | :--- | :--- | :--- |
| **Кол-во процессов ОС** | 1 | 1 | Много | 1 |
| **GIL блокирует?** | Да | **Нет** | Нет (у каждого свой) | Да (не имеет значения) |
| **Потребление памяти** | Низкое | Низкое | **Очень высокое** | **Минимальное** |
| **Идеальный Use Case** | Блокирующий I/O (Selenium) | CPU-bound & I/O | Полная изоляция (pytest-xdist) | Массовый I/O (API, WebSockets) |
| **Сложность дебага** | Высокая (Race conditions) | Очень высокая | Средняя | Средняя |

### Часть 6. Best Practices & Bad Practices (SDET Контекст)

#### ❌ Bad Practices (Архитектурные ошибки)
1.  **Блокировка Event Loop (Blocking the Loop):**
    Самая частая ошибка при переходе на `asyncio`. Использование синхронной библиотеки `requests` внутри `async def` функции. Синхронный код заблокирует *весь* поток, и ни один другой асинхронный тест не выполнится, пока `requests` не получит ответ.
    *Решение:* Использовать `httpx.AsyncClient` или `aiohttp`.
2.  **`time.sleep()` в асинхронных тестах:**
    Он замораживает весь процесс. Нужно использовать **исключительно** `await asyncio.sleep()`.
3.  **Создание 1000 процессов `multiprocessing` для API тестов:**
    Попытка распараллелить легкие сетевые запросы через процессы приведет к тому, что 95% процессорного времени уйдет на создание самих процессов ОС и переключение контекста (Context Switching), а не на тестирование.

#### ✅ Best Practices (Стандарты 2026)
1.  **Гибридный подход (`run_in_executor`):**
    Если внутри быстрого асинхронного API-фреймворка вам вдруг нужно выполнить тяжелую синхронную задачу (например, сгенерировать сложный RSA-ключ через Legacy библиотеку), не ломайте Event Loop. Вынесите эту задачу в отдельный поток:
    ```python
    import asyncio
    
    async def test_crypto_api():
        loop = asyncio.get_running_loop()
        # Выполняем тяжелый синхронный код в пуле потоков (не блокируя Event Loop)
        token = await loop.run_in_executor(None, legacy_heavy_crypto_function)
        # Отправляем асинхронный запрос
        response = await async_client.post("/auth", json={"token": token})
    ```
2.  **Ограничение конкурентности (Concurrency Limiting):**
    При использовании `asyncio.gather()` для запуска 5000 тестов, вы можете устроить DDoS-атаку на собственный тестовый стенд. Senior SDET всегда использует `asyncio.Semaphore` для контроля нагрузки:
    ```python
    sem = asyncio.Semaphore(100) # Не более 100 одновременных соединений
    
    async def bounded_api_call(client, url):
        async with sem:
            return await client.get(url)
    ```
3.  **Управление Free-threaded окружениями через `uv`:**
    В 2026 году стандартом управления пакетами стал написанный на Rust менеджер `uv` (от Astral). При настройке CI/CD пайплайнов, Senior SDET использует его для явного тестирования совместимости с No-GIL:
    `uv run --python 3.14t pytest tests/` — это гарантирует, что ваши автотесты работают в высокопроизводительном истинно-многопоточном режиме без влияния GIL.

---

### Резюме
В 2026 году Senior SDET больше не скован рамками GIL благодаря Python 3.14t. Однако **Asyncio** остается абсолютным королем для API и E2E тестирования (где 99% времени уходит на I/O). Истинная **Многопоточность (Threading без GIL)** теперь применяется для тяжелой подготовки тестовых данных и криптографии, а **Многопроцессность** используется только там, где требуется 100% изоляция памяти при падениях (как в `pytest-xdist` workers). Умение комбинировать эти 3 инструмента — признак архитектурной зрелости инженера.
