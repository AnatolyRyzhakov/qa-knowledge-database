# 📘 Глава 5. Управление тестовыми данными (Test Data Management)
## Тема 5.1. Стратегии подготовки: Data Seeding через API vs Базы данных

### Предисловие: Test Data Management (TDM)
Переход к **Test Data Management (TDM)** — это момент, когда фреймворк из "набора скриптов" превращается в Enterprise-решение. В 2026 году главная причина нестабильных (flaky) тестов в CI/CD — это конфликты из-за общих или захардкоженных тестовых данных [4][6].

### Введение: Контекст стандартов (ISO 25010 и ISTQB)
Согласно стандарту **ISO/IEC 25010:2023**, качество программного обеспечения оценивается по критериям **Reliability** (Надежность) и **Security** (Безопасность). Использование "грязных" дампов из Production-баз с реальными данными пользователей (PII) в тестах строго запрещено законами (GDPR) и стандартами безопасности [5]. 

В руководстве **ISTQB Advanced Level Test Automation Engineer** подчеркивается, что автотесты должны быть **независимыми (Independent)** [2]. Каждый тест должен сам подготовить для себя уникальные данные (Data Seeding) перед выполнением и очистить их после себя (Teardown) [6]. В 2026 году для "посева" данных (Seeding) SDET использует два основных архитектурных подхода: прямые инъекции в БД и генерацию через API.

---

### Часть 1. Стратегия 1: Data Seeding через Базы Данных (Direct DB Injection)

Этот подход подразумевает прямое подключение тестового фреймворка к базе данных (PostgreSQL, MongoDB) и выполнение SQL-скриптов или использование ORM (Object-Relational Mapping) для вставки нужных записей[1][6].

#### Архитектура в 2026 году (Database-per-Test)
В современных CI/CD конвейерах SDET больше не подключается к "общей" тестовой базе данных, так как параллельные потоки `xdist` перезапишут данные друг друга [3]. Стандартом является паттерн **Database-per-Test** (База данных на каждый тест) или **Ephemeral Databases** (Эфемерные БД) [3][4].
Используя инструмент **Testcontainers** (или Docker Compose), автотест поднимает пустую БД в изолированном контейнере за миллисекунды, накатывает туда SQL-миграции, заливает стартовые данные (`INSERT INTO...`), прогоняет проверки и уничтожает контейнер [4].

**Плюсы:**
*   **Сверхскорость:** Прямая вставка тысяч записей (Bulk Insert) в БД происходит в 10–50 раз быстрее, чем создание тех же записей через HTTP API [1].
*   **Тестирование "Негативных" состояний:** Вы можете искусственно создать в БД поврежденные (corrupted) или пограничные данные, которые невозможно создать через API (так как API имеет встроенную валидацию, блокирующую некорректный ввод).

**Минусы:**
*   **Хрупкость (Coupling):** Тесты жестко привязываются к схеме базы данных. Если разработчик переименует колонку `user_name` в `first_name`, ваши SQL-инъекции упадут, хотя бизнес-логика продукта не изменилась [6].
*   **Обход бизнес-логики:** Прямая запись в БД не триггерит внутренние процессы приложения (например, не отправляется email-уведомление о регистрации, не обновляется кэш Redis), что делает тестовое состояние "нереалистичным".

### Часть 2. Стратегия 2: Data Seeding через API (Programmatic Seeding)

В этой стратегии автотест (даже если это UI-тест) использует **внутренние или публичные API-эндпоинты** приложения для создания тестового юзера, корзины и заказа на этапе `Setup` [2][4].

**Пример (Pytest + Playwright):**
```python
import pytest

@pytest.fixture
async def seeded_user(api_client, fake_data):
    # Setup: Создаем пользователя через API (Programmatic Seeding)
    payload = {"email": fake_data.email(), "password": "Secure123!"}
    response = await api_client.post("/v1/users", json=payload)
    user_id = response.json()["id"]
    
    yield payload # Передаем данные UI-тесту для логина
    
    # Teardown: Удаляем пользователя через API
    await api_client.delete(f"/v1/users/{user_id}")
```

**Плюсы:**
*   **Интеграционная надежность:** API гарантирует, что данные созданы с соблюдением всех бизнес-правил продукта. Обновятся кэши, сработают триггеры [4].
*   **Устойчивость к рефакторингу:** Тесты не ломаются при изменениях внутренней структуры таблиц БД [6].

**Минусы:**
*   **Сложность оркестрации:** Если для тестирования "Оплаты заказа" вам нужно сначала создать Юзера $\rightarrow$ создать Магазин $\rightarrow$ добавить Товар $\rightarrow$ собрать Корзину, API Seeding потребует выполнения 5–10 последовательных HTTP-запросов перед *каждым* тестом, что замедляет пайплайн.

### Часть 3. Матрица выбора для Senior SDET (Стек 2026)

Хороший архитектор не выбирает один инструмент навсегда, а комбинирует их в зависимости от уровня тестирования (Пирамида тестов) [4][6]:

| Уровень тестов | Идеальная стратегия Seeding'а | Обоснование (Best Practice 2026) |
| :--- | :--- | :--- |
| **Unit / Component** | **In-memory Mocks / SQLite** | Никаких внешних зависимостей. Данные "сеются" в память на миллисекунды [4]. |
| **Integration (API)** | **Testcontainers + DB Injection** | На этом уровне проверяются контракты и микросервисы. Поднимается свежая БД в Docker, данные заливаются SQL-скриптами для проверки сложной логики бэкенда [3][4]. |
| **E2E (Web / Mobile)** | **API Seeding** | UI-тесты самые медленные. Использовать UI для создания данных (кликать "Добавить") — антипаттерн. Данные мгновенно создаются через API, а UI-тест проверяет только финальный шаг [4][6]. |

### Часть 4. Best Practices & Anti-patterns (Эпоха CI/CD)

#### ❌ Anti-patterns (Красные флаги на ревью)
1.  **Отсутствие стратегии (Shared Data Dependency):** Использование "вечных" аккаунтов (`testuser1@mail.com`), зашитых в конфигурацию. В параллельных прогонах `pytest-xdist` такие тесты будут перехватывать данные друг у друга, вызывая случайные падения (Flaky tests) [3][6].
2.  **Создание данных через UI:** Если вы тестируете функционал "Удаление из корзины", использование Selenium/Playwright для шагов "Авторизация $\rightarrow$ Поиск товара $\rightarrow$ Добавление в корзину" занимает 15 секунд. Если этот сетап упадет, тест "Удаление" не выполнится (False Negative). Создавайте корзину через API за 100 мс!
3.  **Использование Production-дампов без маскирования:** Нарушение комплаенса. Если вы используете дампы, они обязаны проходить Data Masking (анонимизацию PII-данных) перед попаданием в тестовую БД [5].

#### ✅ Best Practices (Стандарты 2026)
1.  **Synthetic Data Generation:** Senior SDET больше не пишет `{"name": "test"}`. Используются библиотеки искусственной генерации (например, `Faker`, `Bogus` или встроенные AI-агенты) [3][6]. Это гарантирует, что каждый тестовый прогон будет использовать уникальные email-ы и UUID, предотвращая конфликты уникальных ключей в БД.
2.  **Idempotent Teardown (Идемпотентная очистка):** Очистка данных должна выполняться надежно. Если API Seeding не смог удалить созданного пользователя в конце теста, в БД копится "мусор". SDET должен реализовывать механизмы Garbage Collection (например, скрипты, которые каждую ночь удаляют из тестовых сред все сущности с тегом `created_by: automation`) [2].
3.  **Использование "Golden Datasets" (Золотые наборы):** Для нагрузочного (Performance) тестирования используются подготовленные, анонимизированные снимки (Snapshots) баз данных, которые монтируются в контейнеры при старте конвейера [4].
