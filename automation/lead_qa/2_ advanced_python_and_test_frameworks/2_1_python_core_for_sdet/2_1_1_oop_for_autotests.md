## 🏛️ 1. Абстрактные классы (ABC): Жесткие контракты

**Проблема:** У тебя в фреймворке есть 10 разных API-клиентов (AuthAPI, BillingAPI, UserAPI и т.д.). Приходит новый автоматизатор Вася, пишет 11-й клиент, но забывает добавить туда метод авторизации или проверку `healthcheck`. Код ревьюить сложно, стандартов нет.

**Решение:** Использование модуля `abc` (Abstract Base Classes). Абстрактный класс — это **чертеж (контракт)**. Он запрещает создавать свои экземпляры напрямую и *заставляет* всех наследников реализовать определенные методы.

### 💻 Пример: Контракт для API-клиентов (с фишкой Python 3.12+)

В современном Python мы используем декоратор `@override` из модуля `typing`, чтобы линтер (`Mypy`) сразу бил по рукам, если мы ошиблись в названии переопределяемого метода.

```python
from abc import ABC, abstractmethod
from typing import override
import httpx

# 1. АБСТРАКТНЫЙ КЛАСС (Интерфейс/Контракт)
class AbstractServiceApi(ABC):
    
    @abstractmethod
    def ping(self) -> bool:
        """Каждый микросервис обязан уметь проверять свое здоровье"""
        pass
        
    @abstractmethod
    def get_version(self) -> str:
        pass

# 2. РЕАЛИЗАЦИЯ (Конкретный клиент)
class BillingApiClient(AbstractServiceApi):
    def __init__(self, base_url: str):
        self.client = httpx.Client(base_url=base_url)

    # @override гарантирует, что мы точно переопределяем метод родителя (PEP 698)
    @override
    def ping(self) -> bool:
        response = self.client.get("/health")
        return response.status_code == 200

    @override
    def get_version(self) -> str:
        return self.client.get("/version").json().get("version", "unknown")

# Попытка сделать obj = AbstractServiceApi() вызовет ошибку TypeError!
```

## 🧬 2. Наследование vs Композиция

В ООП есть золотое правило, которое часто нарушают автоматизаторы: **"Композиция предпочтительнее наследования" (Composition over Inheritance)**.

**Антипаттерн в тестах:** Выстраивание глубоких деревьев наследования. 
`BaseTest -> ApiTest -> AuthApiTest -> AdminAuthApiTest -> SuperAdminAuthApiTest`. 
Когда в `BaseTest` добавляется новый метод (например, очистка БД), он ломает логику в `SuperAdminAuthApiTest`, и найти причину невозможно (проблема "Fragile Base Class").

**Best Practice:** В 2026 году наследование используется **максимум на 1-2 уровня** вглубь. Если классу нужны дополнительные функции — он получает их через компоненты (композицию) или **Миксины**.

## 🍹 3. Миксины (Mixins): Примеси поведения

**Что такое Mixin?** Это класс, который не предназначен для самостоятельного создания. Его задача — "подмешать" (добавить) определенное поведение в другой класс. 

**Правила хорошего Миксина:**
1. Он **не должен иметь состояния** (никаких методов `__init__` и `self.x = ...`, чтобы не ломать `__init__` базового класса).
2. Он решает только одну узкую задачу (SRP).

В автотестах миксины идеальны для добавления:
* Логирования (Allure-степов).
* Механизмов Retry (повторных попыток).
* Парсинга JWT токенов.

### 💻 Пример: Паттерн Mixin для логирования и повторных запросов

```python
import time
import httpx

# --- МИКСИНЫ ---

class AllureLoggerMixin:
    """Миксин добавляет возможность логировать шаги в Allure"""
    def log_step(self, message: str) -> None:
        # В реальности тут будет allure.step / allure.attach
        print(f"[ALLURE STEP] {message}")

class RetryMixin:
    """Миксин добавляет возможность делать retry при 5xx ошибках"""
    def send_with_retry(self, action: callable, retries: int = 3) -> httpx.Response:
        for attempt in range(retries):
            response = action()
            if response.status_code < 500:
                return response
            print(f"[RETRY] Server error, attempt {attempt + 1} of {retries}")
            time.sleep(1)
        raise RuntimeError("Service is dead after retries")


# --- БАЗОВЫЙ КЛАСС КЛИЕНТА ---
class BaseApiClient:
    def __init__(self, client: httpx.Client):
        self.client = client


# --- ИТОГОВЫЙ КЛАСС С МНОЖЕСТВЕННЫМ НАСЛЕДОВАНИЕМ ---
# Мы "подмешиваем" в клиент логирование и ретраи
class UserApiClient(BaseApiClient, AllureLoggerMixin, RetryMixin):
    
    def get_user(self, user_id: int) -> httpx.Response:
        self.log_step(f"Получение данных пользователя {user_id}")
        
        # Используем метод из RetryMixin, а внутри делаем вызов из BaseApiClient
        return self.send_with_retry(
            action=lambda: self.client.get(f"/users/{user_id}")
        )
```

**Почему это круто?** Если завтра тебе понадобится `PaymentClient` БЕЗ ретраев, ты просто не унаследуешь его от `RetryMixin`. Твой код гибкий как конструктор Lego.

## 🧠 MRO: Главный вопрос на собеседовании

Когда ты используешь множественное наследование (как в примере выше с миксинами), Python должен понимать, в каком порядке искать методы, если они называются одинаково. Этот порядок называется **MRO (Method Resolution Order)**.

Алгоритм поиска в современном Python (алгоритм C3-линеаризации): **Слева направо, снизу вверх**.

```python
class A:
    def process(self): print("A")

class Mixin1(A):
    def process(self): print("Mixin1")

class Mixin2(A):
    def process(self): print("Mixin2")

class MyTest(Mixin1, Mixin2):
    pass

# Что выведет MyTest().process()?
# Порядок MRO: MyTest -> Mixin1 -> Mixin2 -> A -> object
# Ответ: "Mixin1", так как он стоит левее.
```

## 🎤 Как это спросят на собеседовании

> **Вопрос 1: "В чём отличие абстрактного класса от обычного базового класса?"**
* **Ответ Лида:** "Абстрактный класс (`ABC`) служит контрактом. В отличие от обычного класса, его невозможно инстанцировать (создать объект напрямую). Он гарантирует, что все классы-наследники обязательно реализуют методы, помеченные декоратором `@abstractmethod`. Это защищает архитектуру фреймворка от человеческих ошибок: если автоматизатор забудет написать нужный метод, скрипт упадет еще на этапе загрузки, а не в середине 2-часового тест-рана."

> **Вопрос 2: "Объясни, что такое MRO и зачем тебе, как QA, понимать его работу?"**
* **Ответ Лида:** "MRO (Method Resolution Order) — это порядок, в котором Python ищет методы и атрибуты при множественном наследовании. В автоматизации мы часто используем паттерн Mixin (например, для добавления логирования БД или перехвата сети в Playwright). Если в двух миксинах окажется метод с одинаковым именем (например, `setup()`), понимание алгоритма (слева-направо) MRO убережет меня от непредвиденного поведения и поможет правильно расположить классы в кортеже наследования."

> **Вопрос 3: "Ты проектируешь UI фреймворк. Будешь ли ты делать класс `Header` наследником класса `BasePage`? Почему?"**
* **Ответ Лида:** "Нет, это нарушение принципа логического наследования и антипаттерн. Класс `Header` (Шапка сайта) — это не страница, это её компонент. Наследование подразумевает отношение **'IS A' (ЯВЛЯЕТСЯ)**. Шапка не является страницей. Я буду использовать **Композицию (отношение 'HAS A' / 'СОДЕРЖИТ')**: класс конкретной страницы будет просто создавать внутри себя экземпляр компонента `Header`."
