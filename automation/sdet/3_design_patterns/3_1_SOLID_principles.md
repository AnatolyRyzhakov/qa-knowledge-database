# 📘 Глава 3. Проектирование и Design Patterns
## Тема 3.1. Принципы SOLID в автоматизации (Разделение слоев UI, API, Infra)

### Введение: Архитектура как фундамент (Стандарты 2026)
Согласно **ISTQB Test Automation Engineer Syllabus**, хорошей практикой является проектирование TAF как независимого программного продукта с собственным жизненным циклом [3]. Современный подход (Enterprise Blueprint 2026) диктует отказ от монолитных скриптов в пользу модульной, слоистой архитектуры [4]. 

В правильно спроектированном фреймворке изменения в UI (например, смена фреймворка фронтенда на React) не должны ломать логику API-клиента, а обновление базы данных не должно требовать переписывания шагов (Steps) самого теста [3][4].

---

### Часть 1. Разделение слоев (Layered Architecture)
Прежде чем применять SOLID, фреймворк необходимо логически разделить (Separation of Concerns). В 2026 году индустриальным стандартом является трехслойная (или четырехслойная) модель [3][4]:

1.  **Test Layer (Слой тестов):** Содержит исключительно бизнес-логику проверок. Здесь находятся Assertions (проверки) и вызовы методов из нижних слоев. Тест *никогда* не должен знать, какой CSS-селектор используется для кнопки.
2.  **Business / Domain Layer (Слой бизнес-логики):** Включает Page Objects (для UI) и API-клиенты (для бэкенда). Этот слой переводит бизнес-шаг (например, `login_user()`) в технические команды (кликнуть, отправить POST-запрос).
3.  **Core / Infrastructure Layer (Инфраструктурный слой):** Низкоуровневая обертка над инструментами. Здесь живут фабрики WebDriver, клиенты баз данных, парсеры конфигураций и утилиты для работы с CI/CD артефактами.

### Часть 2. Принципы SOLID глазами Senior SDET

Принципы SOLID, сформулированные Робертом Мартином, работают в автоматизации так же эффективно, как и в backend-разработке [1][5].

#### 1. S - Single Responsibility Principle (Принцип единственной ответственности)
*У класса должна быть только одна причина для изменения.* [1][5]

**Применение в TAF:**
Page Object Model (POM) — классический пример SRP. Класс `LoginPage` должен отвечать **только** за элементы и действия на странице авторизации [1].
*   ❌ **Bad Practice:** Помещение в `LoginPage` методов для генерации тестового пользователя через БД или логики конфигурации драйвера.
*   ✅ **Best Practice:** Создание отдельных классов: `LoginPage` (для UI), `UserDBRepository` (для данных) и `DriverFactory` (для управления браузером). Если меняется схема БД, вы правите только репозиторий; UI-классы остаются нетронутыми [4].

#### 2. O - Open/Closed Principle (Принцип открытости/закрытости)
*Программные сущности должны быть открыты для расширения, но закрыты для модификации.* [1][2]

**Применение в TAF:**
Вы должны иметь возможность добавить новую функциональность (например, поддержку нового браузера или нового метода авторизации), не переписывая существующий базовый код.
*   ✅ **Best Practice:** Использование паттерна Strategy (Стратегия). Допустим, ваше API поддерживает авторизацию через JWT и OAuth. Вместо того чтобы писать огромный `if/else` в базовом классе `APIClient`, вы создаете интерфейс `AuthStrategy` и реализуете два отдельных класса: `JWTAuth` и `OAuth` [4]. Базовый клиент принимает стратегию динамически.

#### 3. L - Liskov Substitution Principle (Принцип подстановки Барбары Лисков)
*Объекты в программе могут быть заменены их наследниками без изменения свойств программы.* [1][2]

**Применение в TAF:**
Тесты не должны зависеть от конкретной реализации инфраструктуры. Если базовый класс требует определенного поведения, все его наследники обязаны это поведение сохранять, не сужая его [1].
*   ✅ **Best Practice:** Если у вас есть абстрактный класс `BaseDatabaseClient` с методом `connect()`, то и `PostgresClient`, и `MongoDBClient` должны корректно реализовывать этот метод, не выбрасывая неожиданных исключений и возвращая ожидаемый статус. Автотест, использующий `BaseDatabaseClient`, должен работать одинаково при подмене конкретной БД под капотом.

#### 4. I - Interface Segregation Principle (Принцип разделения интерфейса)
*Клиенты не должны зависеть от интерфейсов, которые они не используют.* [1][2]

**Применение в TAF:**
Не создавайте "God Objects" (Божественные объекты). Часто джуниоры создают гигантский класс `BaseTest` или `Helper`, в котором есть методы для UI (`click`), API (`send_get`) и БД (`execute_query`). Тест, проверяющий только API, будет вынужден наследовать тяжеловесные зависимости UI-драйвера [5].
*   ✅ **Best Practice:** Разделяйте интерфейсы по ролям (Role-based interfaces). Создайте `UIInteractionHelper`, `APIRequestHelper` и `DatabaseHelper`. В Python это часто реализуется через модульную систему или Mixins (примеси) [5].

#### 5. D - Dependency Inversion Principle (Принцип инверсии зависимостей)
*Модули верхних уровней не должны зависеть от модулей нижних уровней. Оба типа должны зависеть от абстракций.* [1][2][5]

**Применение в TAF:**
Это краеугольный камень масштабируемости. Класс Page Object не должен сам создавать (инстанцировать) браузер [5].
*   ❌ **Bad Practice:** Жесткое связывание (Hard coupling).
    ```python
    class HomePage:
        def __init__(self):
            # Нарушение DIP! Класс сам решает, какой браузер ему нужен.
            self.driver = webdriver.Chrome() 
    ```
*   ✅ **Best Practice:** Внедрение зависимостей (Dependency Injection - DI). Класс должен получать уже готовый драйвер (абстракцию) извне.
    ```python
    class HomePage:
        def __init__(self, driver: WebDriver):
            # Page Object не знает, Chrome это, Safari или Playwright.
            # Он просто работает с переданным интерфейсом.
            self.driver = driver 
    ```
    В экосистеме Pytest инверсия зависимостей изящно решается через **Фикстуры (Fixtures)**: конфигурационный слой создает драйвер и передает его (inject) как аргумент в тестовую функцию [5].

### Часть 3. Архитектурные антипаттерны в 2026 году

При проведении Code Review Senior SDET должен немедленно блокировать (Reject) Pull Request, если замечает следующие нарушения:

1.  **Asserts внутри Page Objects (Нарушение SRP):** Page Object должен только выполнять действия и возвращать информацию с экрана. Проверки (`assert`) должны находиться исключительно в слое `Test Layer`. Исключение составляют только проверки загрузки самой страницы в `__init__` [1][4].
2.  **Смешивание API и UI в одном классе (Нарушение SRP и ISP):** Если ваш тест создает пользователя через API (чтобы ускорить предусловие), а затем логинится через UI, методы API не должны жить в классе `LoginPage`. Координировать их взаимодействие должен специальный слой-оркестратор (Orchestrator/Facade) [4].
3.  **Хардкод конфигураций (Нарушение DIP):** Чтение переменных окружения (например, URL стенда) прямо внутри API-клиента. Клиент должен зависеть от абстракции `ConfigManager`, которая передается ему при инициализации [5].

---

### Резюме
Внедрение принципов SOLID превращает набор скриптов в надежный **Test Automation Framework**. Разделение слоев (UI, API, Инфраструктура) обеспечивает изоляцию, а SOLID гарантирует, что изменение в одном слое (например, переход с Selenium на Playwright в инфраструктурном слое) не затронет код самих тестов (Test Layer). Это снижает стоимость поддержки (Maintenance Cost) и позволяет масштабировать автоматизацию на уровне всей компании.
