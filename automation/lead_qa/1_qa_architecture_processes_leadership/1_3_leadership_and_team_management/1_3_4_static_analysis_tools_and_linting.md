# 📂 Module 1.3.4. Статические анализаторы и линтинг как часть культуры (Ruff, Mypy)

Привет! Мы добрались до финала первого модуля. Это мостик между управлением процессами и написанием хардкорного кода. 

На позиции Lead QA тебе предстоит решить одну из самых раздражающих проблем в любой команде — **бесконечные споры на Code Review о форматировании кода** (где ставить пробелы, как сортировать импорты и нужны ли кавычки). Если разработчики тратят время на обсуждение стиля, они сжигают деньги бизнеса.

В 2025–2026 годах концепция звучит так: **Линтинг — это не набор правил, это культура команды, автоматизированная на 100%**. Машина форматирует, Лид ревьюит архитектуру.

Давай разберем два главных инструмента, которые стали абсолютным стандартом индустрии.

## ⚡ 1. Ruff: Конец эпохи Flake8 и Black

Если до 2024 года стандартом де-факто был зоопарк утилит (`Flake8` для поиска ошибок, `Black` для форматирования, `isort` для сортировки импортов и `Bandit` для безопасности), то в 2025–2026 годах индустрию захватил **Ruff**.

**Что такое Ruff?** 
Это сверхбыстрый линтер и форматер для Python, написанный на языке Rust командой Astral.
* **Скорость:** Он работает в 10–100 раз быстрее аналогов. Линтинг фреймворка из 50 000 строк кода занимает миллисекунды.
* **Всё-в-одном:** Ruff заменяет собой более 50 других плагинов и инструментов.
* **Автофикс:** Он не просто говорит, что код плохой, он сам его исправляет (`ruff check --fix`).

**Почему Лид выбирает Ruff?**
Потому что он экономит время в CI/CD пайплайнах. Вместо установки 5 разных библиотек, ты ставишь один бинарник (часто в связке с пакетным менеджером `uv` от той же команды), и твой пайплайн ускоряется на минуты.

## 🛡️ 2. Mypy: Строгая типизация как гарант стабильности

Python — язык с динамической типизацией. Но в больших Enterprise-фреймворках (500+ автотестов, сложные Pydantic-модели, интеграции) динамическая типизация превращается в ад. Ты передаешь в функцию `user_id`, а функция ждет `int`, но получает `str`, и тест падает на 40-й минуте прогона.

Согласно опросам 2025 года (Python Typing Survey), Type Hints (тайп-хинты) стали **обязательным (non-negotiable)** стандартом.

**Что дает Mypy (или его аналог Pyright)?**
Это статический анализатор типов. Он проверяет твой код *до* его запуска. Если функция ожидает `List[str]`, а ты передаешь `dict`, Mypy подсветит ошибку прямо в IDE и не даст закоммитить код.

## 🛠 3. Практика: Настройка культуры (CI/CD + Pre-commit)

Как заставить команду соблюдать эти правила? Лид никогда не просит инженеров запускать линтеры руками. Он встраивает их в процесс.

### Шаг 1. Конфигурация в `pyproject.toml`
Вся конфигурация проекта должна лежать в одном файле. Никаких `setup.cfg` или `.flake8`.

```toml
[tool.ruff]
# Ruff заменяет Black, ставим длину строки 120 (стандарт для автотестов)
line-length = 120
target-version = "py312"

[tool.ruff.lint]
# Включаем правила: E (ошибки), F (Flake8), I (сортировка импортов), N (именование)
select =["E", "F", "I", "N"]

[tool.mypy]
python_version = "3.12"
strict = true # Включаем максимальную строгость типизации!
disallow_untyped_defs = true # Запрещаем функции без типизации
ignore_missing_imports = true # Для библиотек без типов (например, старых модулей)
```

### Шаг 2. Pre-commit hooks (Защита разработчика)
Мы заставляем линтеры работать на ноутбуке инженера в момент команды `git commit`. Если код кривой — коммит отменяется.

Создаем файл `.pre-commit-config.yaml`:
```yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.3.0
    hooks:
      - id: ruff
        args:[ --fix ] # Автоматически исправлять то, что можно
      - id: ruff-format # Форматировать код в стиле Black
  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.9.0
    hooks:
      - id: mypy
        additional_dependencies:[ types-requests, pydantic ]
```

### Шаг 3. Защита в GitLab CI (Fail Fast)
Если кто-то обошел pre-commit (сделал `git commit --no-verify`), CI пайплайн должен убить Merge Request на самой первой стадии.

```yaml
lint_code:
  stage: validate
  image: python:3.12-slim
  script:
    - pip install ruff mypy
    - ruff check .
    - ruff format --check .
    - mypy .
  # Эта джоба упадет за 5 секунд, сэкономив ресурсы сервера!
```

## 🎤 Как это спросят на собеседовании

> **Вопрос 1: "Почему в автотестах так важна строгая типизация (Type Hints)? Ведь Python позволяет писать быстрее без них."**
* **Ответ Лида:** "Быстро пишется только первый месяц. Когда фреймворк разрастается до тысяч строк кода, поддержка динамического кода становится невыносимой. Строгая типизация и `Mypy` дают нам три преимущества: 
1. Мощный автокомплит в IDE (инженер сразу видит, какие методы есть у Page Object или API-клиента). 
2. Самодокументируемость кода (не нужно писать Docstrings с описанием типов аргументов).
3. Shift-Left подход к багам: `Mypy` отлавливает ошибки передачи неверных типов данных на этапе написания кода, а не во время 40-минутного прогона тестов в CI/CD."

> **Вопрос 2: "Ты настроил Ruff и Mypy, но джуны жалуются, что линтеры постоянно ругаются, и они тратят по 20 минут на то, чтобы закоммитить код. Что делать?"**
* **Ответ Лида:** "Проблема в том, что они запускают проверки постфактум. Я проведу воркшоп и настрою им интеграцию линтеров напрямую в IDE (VS Code / PyCharm). `Ruff` будет форматировать код и сортировать импорты автоматически при нажатии `Ctrl+S` (Save), а ошибки типов от `Mypy` будут подсвечиваться красным прямо во время набора текста (как в TypeScript). Линтер должен быть невидимым помощником, а не строгим надзирателем."

> **Вопрос 3: "В чем фундаментальное преимущество Ruff перед связкой Flake8 + Black + isort?"**
* **Ответ Лида:** "Ruff написан на Rust, что делает его на несколько порядков быстрее питоновских аналогов. Кроме скорости, главное преимущество архитектурное: это единый инструмент с одним конфигурационным файлом (`pyproject.toml`). Нам больше не нужно разрешать конфликты между `Black`, который хочет перенести строку, и `Flake8`, который ругается на её длину. В экосистеме 2025–2026 годов Ruff — это абсолютный стандарт для новых проектов."
