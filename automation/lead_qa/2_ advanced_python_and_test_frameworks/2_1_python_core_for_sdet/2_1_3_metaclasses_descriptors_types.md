## 🛡️ 1. Строгая типизация (Typing): Тренды Python 3.12/3.13

В 2026 году динамическая типизация в больших тестовых фреймворках считается техническим долгом. Без нее IDE не может помочь разработчику, а баги выявляются только на 40-й минуте прогона (runtime error), а не во время написания кода (static analysis).

### Паттерн: Duck Typing через `Protocol`
Python использует "Утиную типизацию" (Duck Typing). Если объект крякает как утка, это утка. Но как объяснить это `Mypy`? С помощью `typing.Protocol` (структурная типизация).

Вместо того чтобы заставлять все классы наследоваться от одного интерфейса (номинативная типизация), мы объявляем `Protocol`.

```python
from typing import Protocol
from playwright.sync_api import Locator

# 1. Объявляем Протокол (что должен уметь делать объект)
class Clickable(Protocol):
    def click(self) -> None:
        ...

# 2. Функция ждет ЛЮБОЙ объект, у которого есть метод click()
def submit_form(element: Clickable) -> None:
    element.click()

# Playwright Locator имеет метод click(), поэтому Mypy пропустит этот код без ошибок,
# хотя Locator НИКАК не унаследован от нашего класса Clickable!
button: Locator = page.locator("#submit")
submit_form(button)
```

*(Тренд 3.13: В Python 3.13 добавились фичи вроде `TypeIs` (PEP 742) для более умного и интуитивного сужения типов, что делает Mypy-чеки еще точнее).*

## 🪄 2. Дескрипторы: Умные локаторы в Page Object

**Дескриптор** — это любой объект, в котором реализован хотя бы один из магических методов: `__get__`, `__set__` или `__delete__`. Это механизм, который лежит в основе `property`, `classmethod` и `staticmethod`.

**Как это использует QA Лид?** 
В классическом Page Object мы постоянно пишем дублирующийся код для инициализации локаторов или методов их получения. Дескрипторы (Lazy Locators) позволяют сделать атрибуты страницы "умными" — они вычисляются только в момент обращения к ним.

### 💻 Пример: Smart Descriptor для Playwright (Best Practice 2025/2026)

```python
from playwright.sync_api import Page, Locator

# 1. Пишем Дескриптор
class UIElement:
    def __init__(self, selector: str):
        self.selector = selector

    # Метод перехватывает обращение к атрибуту
    def __get__(self, instance, owner) -> Locator:
        if instance is None:
            return self # Обращение через класс
        
        # instance - это объект страницы (LoginPage). 
        # Мы динамически достаем из него объект браузера (page) и ищем элемент!
        return instance.page.locator(self.selector)

# 2. Применяем в Page Object
class LoginPage:
    # Декларативное описание (очень чисто и читаемо)
    username_input = UIElement("#user-name")
    password_input = UIElement("#password")
    login_button = UIElement("#login-button")

    def __init__(self, page: Page):
        self.page = page  # Экземпляр сохраняется здесь

    def do_login(self):
        # В момент обращения к self.username_input срабатывает __get__ дескриптора!
        self.username_input.fill("admin")
        self.password_input.fill("secret")
        self.login_button.click()
```
*Почему это круто?* Код страницы становится полностью декларативным. Нет "простыни" кода в `__init__`. Локаторы вычисляются **лениво** — только тогда, когда тест до них доходит.

## 🏗️ 3. Метаклассы (Metaclasses) vs `__init_subclass__`

**Метакласс** — это "класс для классов". Точно так же, как класс создает объекты (инстансы), метакласс (по умолчанию это `type`) создает сами классы в памяти. 

В QA метаклассы использовались для:
* Автоматической регистрации плагинов или PageObject'ов в едином реестре.
* Реализации паттерна Singleton (хотя в Pytest это антипаттерн, как мы обсуждали ранее).
* Принудительной проверки, что наследники переопределили нужные методы.

### 🛑 Архитектурный совет Лида: Уход от Метаклассов
В 2026 году использовать метаклассы без крайней нужды — **считается плохим тоном**, так как они усложняют код и вызывают конфликты (Metaclass conflict) при множественном наследовании.

**Современная альтернатива:** Магический метод `__init_subclass__`. Он делает 90% того, для чего раньше нужны были метаклассы, но работает проще и понятнее.

### 💻 Пример: Автоматическая регистрация классов тестовых данных (Modern Way)

Представь, что ты пишешь фреймворк, где каждая модель данных (DTO) должна быть зарегистрирована в фабрике.

```python
class TestDataModel:
    # Реестр всех существующих моделей в фреймворке
    registry: dict[str, type] = {}

    # Этот метод вызывается автоматически каждый раз, 
    # когда кто-то наследуется от TestDataModel!
    @classmethod
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        # Автоматически регистрируем класс по его имени
        cls.registry[cls.__name__] = cls
        print(f"[*] Авто-регистрация модели: {cls.__name__}")


# Автоматизатору нужно просто унаследоваться, магия произойдет сама
class UserPayload(TestDataModel):
    pass

class OrderPayload(TestDataModel):
    pass

# Вывод в консоль в момент инициализации скрипта:
# [*] Авто-регистрация модели: UserPayload
# [*] Авто-регистрация модели: OrderPayload

print(TestDataModel.registry) 
# {'UserPayload': <class '__main__.UserPayload'>, 'OrderPayload': <class '__main__.OrderPayload'>}
```

## 🎤 Как это спросят на собеседовании

> **Вопрос 1: "В чём разница между номинативной (наследование) и структурной (Duck typing) типизацией? Как реализовать структурную в Python?"**
* **Ответ Лида:** "При номинативной типизации (как в Java) мы обязаны явно наследовать класс от интерфейса, чтобы анализатор понял совместимость. В Python это делается через `abc.ABC`. Но Python — динамический язык с Duck Typing. Чтобы типизировать его строго, но без жесткого наследования, мы применяем структурную типизацию через `typing.Protocol`. Мы описываем только нужные методы (сигнатуры) в Протоколе, и любой класс, имеющий такие методы, будет автоматически считаться валидным для `Mypy`."

> **Вопрос 2: "Ты пишешь Page Object. Разработчики постоянно меняют структуру DOM, локаторы ломаются, а элементы появляются на странице не сразу (ленивая загрузка фронта). Как дескрипторы помогут тебе решить эту архитектурную проблему?"**
* **Ответ Лида:** "Дескрипторы идеально подходят для реализации паттерна 'Smart / Lazy Locator'. Если объявлять `locator` внутри `__init__`, он может попытаться найти элемент в момент создания страницы, когда DOM еще не отрисовался. Я создам класс-дескриптор с методом `__get__`. В самом Page Object я опишу поля декларативно. В момент, когда тест обратится к свойству (например, `page.button.click()`), дескриптор перехватит вызов, динамически получит драйвер из экземпляра (через аргумент `instance`) и найдет элемент 'здесь и сейчас'."

> **Вопрос 3: "Зачем нужны метаклассы и почему сейчас их рекомендуют заменять на `__init_subclass__`?"**
* **Ответ Лида:** "Метаклассы (`type`) нужны для перехвата момента создания самого класса (а не его экземпляра). Они полезны для валидации атрибутов класса или авторегистрации (например, ORM-модели в SQLAlchemy). Однако они сложны в отладке и приводят к MRO-конфликтам при множественном наследовании. В современном Python (начиная с 3.6+) для большинства таких задач мы используем `@classmethod def __init_subclass__`. Он решает задачу авторегистрации наследников или проверки их атрибутов максимально элегантно, не ломая MRO."
