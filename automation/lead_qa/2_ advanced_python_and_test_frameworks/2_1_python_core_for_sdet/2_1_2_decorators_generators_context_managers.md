## 🎁 1. Декораторы: Магия оберток (и как её типизировать)

**Декоратор** — это функция, которая принимает другую функцию, расширяет её поведение и возвращает новую функцию, не изменяя исходный код. Под капотом это работает благодаря механизму **замыканий (closures)**.

В автоматизации декораторы — это инструмент №1 для:
*   Реализации паттерна Retry (повторный запуск при падении).
*   Логирования (например, `@allure.step`).
*   Таймаутов.

### 💻 Пример: Декоратор `@retry` уровня Senior (Python 3.12+)

Джуны пишут декораторы без параметров и типов. Лид пишет декораторы с параметрами, используя `functools.wraps` (чтобы не потерять имя функции и докстринги) и современные типы `ParamSpec` и `TypeVar` (чтобы IDE понимала, какие аргументы принимает обернутая функция).

```python
import time
from functools import wraps
from typing import Callable, TypeVar, ParamSpec, Any

# ParamSpec и TypeVar позволяют сохранить сигнатуру декорируемой функции для IDE (Mypy)
P = ParamSpec('P')
R = TypeVar('R')

def retry(attempts: int = 3, delay: float = 1.0) -> Callable[[Callable[P, R]], Callable[P, R]]:
    """Декоратор с параметрами для повторного выполнения функции при ошибках"""
    
    def decorator(func: Callable[P, R]) -> Callable[P, R]:
        @wraps(func) # Обязательно! Сохраняет __name__ и __doc__ исходной функции
        def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
            last_exception: Exception | None = None
            
            for attempt in range(1, attempts + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    print(f"[RETRY] Функция {func.__name__} упала (Попытка {attempt}/{attempts}). Ошибка: {e}")
                    last_exception = e
                    time.sleep(delay)
                    
            # Если все попытки исчерпаны, пробрасываем последнюю ошибку
            raise RuntimeError(f"Все {attempts} попыток провалились") from last_exception
            
        return wrapper
    return decorator

# Применение
@retry(attempts=5, delay=2.0)
def get_flaky_api_data(endpoint: str) -> dict[str, Any]:
    # Представим, что тут httpx.get(), который падает через раз
    raise ConnectionError("Network timeout")
```

## ⚡ 2. Генераторы: Ленивые вычисления и экономия RAM

**Генератор** — это функция, которая возвращает данные не все сразу (как список), а по одному элементу по мере необходимости (лениво). Вместо `return` используется ключевое слово `yield`.

**Зачем это QA?** 
1. Чтение логов: Если тебе нужно пропарсить в автотесте лог-файл K8s размером 10 ГБ, `f.readlines()` убьет оперативную память (OOM). Генератор прочитает файл по одной строке, расходуя 2 КБ памяти.
2. Тестовые данные (Data-Driven Testing): Генерация миллионов уникальных пользователей для нагрузочного теста (Locust / k6).

### 💻 Пример: Генератор большого объема тестовых данных

```python
from typing import Generator
import uuid

def generate_users(count: int) -> Generator[dict[str, str], None, None]:
    """Генератор лениво создает пользователей по одному"""
    for i in range(count):
        # yield приостанавливает выполнение функции и отдает значение
        yield {
            "id": str(uuid.uuid4()),
            "username": f"user_{i}",
            "email": f"tester{i}@company.local"
        }

# Использование
user_gen = generate_users(10_000_000)

# Мы не загрузили в память 10 млн словарей! Мы берем только один, когда нужно:
first_user = next(user_gen) 
second_user = next(user_gen)
```
*Бонус 2026 года:* Асинхронные генераторы (`async yield`) активно используются в `httpx` для стриминга ответов (например, при тестировании скачивания файлов или SSE/WebSockets).

## 🛡️ 3. Контекстные менеджеры: Защита от утечек ресурсов

Если ты открыл файл, соединение с БД или инстанс браузера — ты **обязан** их закрыть. Если тест падает с `AssertionError` до вызова `.close()`, ресурс остается "висеть", создавая утечки памяти (Memory Leaks).

Контекстные менеджеры (оператор `with`) гарантируют, что очистка ресурсов произойдет всегда, даже если внутри блока упало исключение.

Под капотом контекстный менеджер — это класс с "магическими" методами `__enter__` (выполняется на входе) и `__exit__` (выполняется на выходе). Но в современном Python чаще используют декоратор `@contextmanager`.

### 💻 Пример: Асинхронный контекстный менеджер (Тренд 2025/2026)

Так как индустрия перешла на асинхронные микросервисы, от SDET требуют умения писать `async with`. Для этого мы используем `contextlib.asynccontextmanager`.

```python
import asyncio
from contextlib import asynccontextmanager
from typing import AsyncGenerator

class AsyncDbConnection:
    async def connect(self): print("Подключение к БД установлено")
    async def close(self): print("Соединение с БД закрыто")
    async def execute(self, query): print(f"Выполняем запрос: {query}")

@asynccontextmanager
async def db_transaction() -> AsyncGenerator[AsyncDbConnection, None]:
    """Асинхронный контекстный менеджер для безопасной работы с БД"""
    conn = AsyncDbConnection()
    await conn.connect()
    
    try:
        # yield передает управление (и объект) внутрь блока 'async with'
        yield conn  
    except Exception as e:
        print(f"[ROLLBACK] Транзакция отменена из-за ошибки: {e}")
        raise
    finally:
        # Этот код выполнится ВСЕГДА, даже если был assert
        await conn.close()

# Использование в тесте
async def test_db_insert():
    async with db_transaction() as db:
        await db.execute("INSERT INTO users ...")
        assert 1 == 2 # Тест падает, но __exit__ (finally) всё равно закроет соединение!
```
*(Спойлер: Фикстуры PyTest с `yield` работают по такому же принципу контекстного менеджера!)*

## 🎤 Как это спросят на собеседовании

> **Вопрос 1: "Почему при написании декоратора важно использовать `@functools.wraps`?"**
* **Ответ Лида:** "Декоратор по своей сути подменяет оригинальную функцию на функцию `wrapper` (обертку). Если не использовать `@wraps`, мы потеряем метаданные оригинальной функции: её имя (`__name__`), документацию (`__doc__`) и аннотации типов. Это сломает генерацию документации (Sphinx/MkDocs), автокомплит в IDE, а в случае автотестов — Allure-отчеты и плагины Pytest не смогут правильно определить названия тестов."

> **Вопрос 2: "Чем `yield` отличается от `return`? И в каких задачах QA ты бы использовал генератор?"**
* **Ответ Лида:** "`return` полностью завершает выполнение функции и выгружает её из стека вызовов, возвращая финальный результат. `yield` приостанавливает выполнение функции, сохраняя её локальное состояние, и отдает промежуточный результат. При следующем вызове `next()` функция продолжает работу с того же места. 
Я использую генераторы для снижения потребления памяти (Memory Footprint). Например: чтение гигантских CSV-файлов с тестовыми данными или потоковое (stream) чтение логов из микросервиса. Вместо загрузки 1 ГБ данных в список, мы держим в памяти только 1 строку."

> **Вопрос 3: "У тебя в тесте открывается коннект к RabbitMQ. Тест упал посередине, коннект остался открытым. Как правильно это архитектурно предотвратить?"**
* **Ответ Лида:** "Обернуть работу с брокером в Контекстный менеджер. Я реализую класс RabbitClient, где пропишу методы `__enter__` и `__exit__` (или `__aenter__`/`__aexit__` для `aio-pika`). В `__enter__` я буду открывать канал, а в `__exit__` гарантированно закрывать его. Тогда в тесте я буду писать `with RabbitClient() as client:`, и если внутри этого блока произойдет `AssertionError` или любой другой Exception, Python автоматически вызовет `__exit__`, и мы избежим утечки соединений (Connection Leaks)."
