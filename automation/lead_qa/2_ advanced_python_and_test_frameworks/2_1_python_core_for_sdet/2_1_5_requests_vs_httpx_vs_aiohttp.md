# 📂 Module 2.1.5. Работа с HTTP: `Requests` vs `httpx` vs `aiohttp`

Привет! Завершаем глубокое погружение в Core Python темой, которая напрямую влияет на производительность твоего фреймворка. В 2025–2026 годах тестирование микросервисов — это постоянная работа с сетью. Если твой HTTP-клиент работает неэффективно, тесты будут выполняться часами.

На собеседованиях Лидов часто просят аргументировать выбор библиотеки для API-тестирования. Ответить *"Я взял requests, потому что привык"* — значит провалить проверку на знание трендов и архитектуры.

Давай разберем «Большую тройку» HTTP-клиентов и поймем, почему индустрия сделала свой выбор.

---

## ⚖️ 1. Сравнение библиотек (Тренды 2025-2026)

### 🦕 Requests: Старый король
Библиотека `requests` была стандартом де-факто более 10 лет. Её главный плюс — невероятно простой и понятный интерфейс (API).
* **Минусы для 2026 года:** Она строго **синхронная** (блокирующая). Если тебе нужно отправить 1000 запросов, они уйдут строго по очереди. Кроме того, `requests` **не поддерживает HTTP/2**, что делает его аутсайдером при тестировании современных gRPC/REST фасадов, которые мультиплексируют трафик.
* **Вердикт:** Оставляем только для простейших одноразовых скриптов или Legacy-проектов.

### 🏎️ aiohttp: Первопроходец асинхронности
Когда в Python появился `asyncio`, разработчикам понадобился асинхронный клиент. Им стал `aiohttp`. Это мощнейший комбайн, который может быть не только клиентом, но и веб-сервером.
* **Минусы:** У него **другой синтаксис**, не похожий на привычный `requests` (приходится переучивать команду). Он работает *только* в асинхронном режиме (нельзя написать синхронный тест, если очень нужно). 
* **Вердикт:** Отличный выбор для написания высоконагруженных парсеров (Web Scraping) или работы с WebSockets, но для E2E API автотестов он слишком громоздок.

### 🚀 httpx: Современный стандарт (Modern Choice)
В 2025–2026 годах `httpx` стал абсолютным стандартом (де-факто) для новых QA-фреймворков и SDK (его используют OpenAI и Anthropic).
* **Плюсы:** 
  1. Синтаксис на 99% совместим с `requests` (миграция занимает пару часов).
  2. Поддерживает **и синхронный, и асинхронный** режимы в одной библиотеке (`httpx.Client` и `httpx.AsyncClient`).
  3. Нативная поддержка **HTTP/2** (мультиплексирование потоков).
  4. Глубокая интеграция с FastAPI: возможность тестировать код *In-Memory* без поднятия реального Uvicorn сервера.

## 🏗️ 2. Архитектура: Ошибки уровня Junior vs Код уровня Lead

Главная ошибка начинающих автоматизаторов при использовании `requests` или `httpx` — использование методов верхнего уровня (например, `httpx.get()`). 

Каждый раз, когда ты вызываешь `httpx.get(...)`, библиотека:
1. Открывает новое TCP-соединение (Handshake).
2. Выполняет TLS-рукопожатие (если это HTTPS).
3. Отправляет запрос и закрывает соединение.
*На 100 запросах ты потеряешь секунды только на рукопожатиях!*

**Архитектурное решение (Connection Pooling):** Использование сессий (клиентов) через контекстные менеджеры `httpx.AsyncClient()`. Они держат TCP-соединение открытым (Keep-Alive) и переиспользуют его для следующих запросов.

### 💻 Пример: Продвинутая фикстура API-клиента на httpx (PyTest)

Вот так выглядит идеальная инициализация клиента в `conftest.py` для тестирования микросервисов в 2026 году:

```python
import pytest
import httpx
from typing import AsyncGenerator

@pytest.fixture(scope="session")
async def api_client() -> AsyncGenerator[httpx.AsyncClient, None]:
    """
    Глобальная сессия для API тестов. 
    Экономит ресурсы за счет Connection Pooling.
    """
    # Включаем HTTP/2 и настраиваем строгие таймауты (чтобы тесты не висли вечно)
    timeout = httpx.Timeout(
        timeout=10.0,      # Общий таймаут
        connect=2.0,       # Таймаут на TCP handshake
        pool=2.0           # Ожидание свободного коннекта в пуле
    )
    
    # Настраиваем лимиты пула соединений (важно для параллельного запуска тестов!)
    limits = httpx.Limits(max_keepalive_connections=50, max_connections=100)

    # Инициализируем клиент один раз на всю сессию
    async with httpx.AsyncClient(
        base_url="https://api.microservice.com/v1",
        http2=True,
        timeout=timeout,
        limits=limits,
        headers={"X-Test-Env": "QA"}
    ) as client:
        
        print("\n[INIT] Подняли AsyncClient (HTTP/2 enabled)")
        yield client # Отдаем клиент в автотесты
        print("\n[TEARDOWN] Закрыли пул TCP-соединений")
```

## ⚡ 3. In-Memory тестирование FastAPI (Хак для SDET)

Если твой бэкенд написан на FastAPI, `httpx` дает QA-инженеру суперсилу. Тебе больше не нужно поднимать базу, Docker и Uvicorn, чтобы протестировать логику роутеров. 
Ты можешь скормить объект `app` напрямую в `httpx.AsyncClient(app=app)`. 

Запрос не пойдет через сетевую карту компьютера! Он напрямую вызовет Python-код бэкенда, эмулируя HTTP-протокол. Такие "E2E-интеграционные" тесты выполняются за **миллисекунды**.

```python
from fastapi import FastAPI
import httpx
import pytest

# Твое реальное приложение импортируется из бэкенда
from myapp.main import app 

@pytest.mark.asyncio
async def test_fastapi_in_memory():
    # Запросы идут напрямую в ASGI приложение, в обход сети!
    async with httpx.AsyncClient(app=app, base_url="http://testserver") as client:
        response = await client.post("/users", json={"name": "Alice"})
        
        assert response.status_code == 201
```

## 🎤 Как это спросят на собеседовании

> **Вопрос 1: "Почему для нового проекта в 2026 году ты выберешь `httpx`, а не `requests`?"**
* **Ответ Лида:** "Первая причина — нативная поддержка асинхронности, без которой невозможно конкурентно тестировать Event-Driven микросервисы. Вторая — поддержка HTTP/2, что позволяет использовать мультиплексирование потоков. Третья — он сохраняет привычный синтаксис `requests`, поэтому порог входа для мануальных QA, переходящих в автоматизацию, остается минимальным. И наконец, `httpx` де-факто интегрирован во все современные Python-фреймворки типа FastAPI для написания TestClient."

> **Вопрос 2: "Ты написал скрипт: `for i in range(100): httpx.get(url)`. Он работает очень медленно. В чём архитектурная ошибка и как её исправить?"**
* **Ответ Лида:** "Ошибка в том, что `httpx.get()` на каждой итерации открывает и закрывает TCP-сокет и выполняет заново SSL-хендшейк. Чтобы это исправить, нужно использовать паттерн Connection Pooling. Я оберну цикл в контекстный менеджер `with httpx.Client() as client:` и внутри буду вызывать `client.get()`. Соединение будет установлено один раз и переиспользовано 100 раз (Keep-Alive). Для максимальной скорости я перепишу это на `AsyncClient` и использую `asyncio.TaskGroup` для отправки всех 100 запросов конкурентно."

> **Вопрос 3: "Что такое мультиплексирование в HTTP/2 и как оно помогает в автотестах?"**
* **Ответ Лида:** "В HTTP/1.1 (который использует `requests`) мы страдаем от проблемы Head-of-Line Blocking: в рамках одного TCP-соединения мы можем отправить следующий запрос только после получения ответа на предыдущий. Если сервер "думает" над первым запросом, остальные ждут в очереди. HTTP/2 мультиплексирует данные: он разбивает запросы на бинарные фреймы и отправляет их параллельно по ОДНОМУ TCP-соединению. При включении `http2=True` в `httpx`, наши высоконагруженные тесты отрабатывают быстрее, не исчерпывая лимиты открытых портов (сокетов) на CI-сервере."
