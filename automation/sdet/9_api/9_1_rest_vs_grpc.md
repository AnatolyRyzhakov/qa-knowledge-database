# 📘 Глава 9. Архитектура API и Микросервисы
## Тема 9.1. REST vs gRPC: Выбор протокола, идемпотентность HTTP-методов

### Введение: Контекст стандартов (ISO 25010 и ISTQB)
Согласно стандарту **ISO/IEC 25010:2023**, качество распределенных систем оценивается через характеристики **Interoperability (Совместимость)** и **Reliability (Надежность)**. В микросервисной архитектуре тестировщик проверяет не только бизнес-логику, но и сетевые контракты между компонентами [7][8]. 
Перед архитектором и QA-лидом встает важнейший выбор протокола взаимодействия: классический REST или высокопроизводительный gRPC [2][3].

---

### Часть 1. REST vs gRPC: Битва протоколов в 2026 году

В современных системах выбор между REST и gRPC — это поиск компромисса между универсальностью (Accessibility) и ультимативной производительностью (Performance) [2][5].

#### 1. REST (Representational State Transfer)
Архитектурный стиль, использующий стандартные методы HTTP (GET, POST, PUT, DELETE) и текстовые форматы (в 99% случаев JSON) [3].
*   **Плюсы:** Идеальная читаемость человеком (Human-readable), нативная поддержка абсолютно во всех браузерах, простота отладки через Chrome DevTools [3][5].
*   **Минусы:** JSON — очень "тяжелый" текстовый формат. Он требует много процессорного времени на сериализацию и десериализацию, не поддерживает нативную строгую типизацию [3].
*   **Где применяется:** Публичные API (Public-facing APIs), интеграция с веб-браузерами и мобильными клиентами [2][3].

#### 2. gRPC (gRPC Remote Procedure Calls)
Фреймворк от Google, работающий поверх протокола HTTP/2. Данные передаются в строгом бинарном формате **Protocol Buffers (Protobuf)** [3].
*   **Плюсы:** Сверхскорость (в 5–10 раз быстрее REST) за счет бинарного сжатия и мультиплексирования. Наличие строгих контрактов (`.proto` файлов). Поддержка двунаправленного потокового вещания (Bidirectional Streaming) [2][5].
*   **Минусы:** Бинарный код невозможно прочитать глазами без расшифровки (десериализации). Не поддерживается браузерами напрямую (требует gRPC-Web прокси) [2][5].
*   **Где применяется:** Внутреннее общение между микросервисами (Internal Service-to-Service communication) в High-Load системах [3][5].

#### 🛠️ Специфика тестирования (SDET View):
Если проект использует gRPC, классическая библиотека `requests` в Python не поможет. Senior SDET использует инструменты вроде `grpcurl` или компилирует `.proto` файлы в Python-заглушки (Client Stubs), чтобы вызывать методы удаленного сервера как локальные функции [6].

### Часть 2. Идемпотентность HTTP-методов (Критический навык SDET)

На архитектурных интервью вопрос об идемпотентности задают практически всегда.
**Идемпотентность (Idempotency)** означает, что многократное выполнение одного и того же запроса дает *ровно тот же самый результат (и оставляет сервер в том же состоянии)*, что и однократное выполнение[1][4].

В распределенных микросервисах сетевые сбои случаются постоянно. Если клиент не получил ответ по таймауту, он автоматически повторяет (Retry) запрос [1]. Идемпотентность гарантирует, что эти автоматические повторы не сломают систему (например, не спишут деньги с карты клиента дважды) [1][4].

#### Матрица Идемпотентности (По стандарту RFC 9110):
1.  **`GET`, `HEAD`, `OPTIONS` (Safe & Idempotent):** Безопасные методы. Они только читают данные, не меняя состояние базы данных. Запрашивайте их хоть миллион раз — состояние системы не изменится [1][4].
2.  **`PUT`, `DELETE` (Idempotent):** 
    *   `PUT` полностью перезаписывает ресурс. Если вы отправите `PUT /user/1 {"name": "Alex"}` десять раз, имя останется "Alex". Состояние не меняется после первой успешной попытки [1][4].
    *   `DELETE /user/1` удалит пользователя. При повторных запросах сервер вернет `404 Not Found` (или `200 OK`, в зависимости от реализации), но *состояние системы останется прежним* (пользователь отсутствует). Следовательно, метод идемпотентен [1].
3.  **`POST`, `PATCH` (NOT Idempotent):**
    *   Каждый запрос `POST /orders` создает *новый* заказ. Отправите 5 раз — получите 5 новых заказов и спишете деньги 5 раз. Это не идемпотентно по умолчанию [1][4].

### Часть 3. Как сделать POST идемпотентным (Тренд 2026)

Так как микросервисам необходимо безопасно повторять `POST`-запросы (например, платежные транзакции), в индустрии утвердился стандарт **Идемпотентных Ключей (Idempotency Keys)** [1][4].

**Архитектура защиты от дублей:**
1.  Клиент (или ваш автотест) генерирует уникальный UUID и добавляет его в HTTP-заголовок `Idempotency-Key` [1].
2.  Сервер принимает запрос и проверяет ключ в кэше (например, в Redis).
3.  Если ключа нет, сервер выполняет операцию (списывает деньги) и сохраняет ключ вместе с успешным ответом в кэш.
4.  Если из-за сбоя сети клиент отправляет тот же `POST` с тем же `Idempotency-Key` второй раз, сервер видит ключ в кэше, *не выполняет списание*, а просто возвращает закэшированный ответ оригинального запроса [1][4].

### Часть 4. Best Practices & Anti-patterns (SDET Контекст)

#### ❌ Anti-patterns (Красные флаги на ревью)
1.  **Тестирование gRPC как REST:** Игнорирование `.proto` файлов. В gRPC контракты (`.proto`) — это единственный источник истины (Single Source of Truth) [6]. Senior SDET должен настроить пайплайн так, чтобы тесты падали на этапе генерации стабов, если разработчик сломал обратную совместимость контракта (Contract Validation) [6][7].
2.  **Использование `GET` для изменения состояния:** Написание API, где `GET /api/users/delete?id=5` удаляет данные. Это катастрофическое нарушение стандартов, которое приведет к тому, что поисковые роботы (crawlers) или механизмы предзагрузки браузера случайно удалят вашу базу данных [4].
3.  **Отсутствие тестов на Retries:** Тестирование POST-методов только "счастливым путем" (Happy Path). Если API предназначено для распределенных систем, SDET обязан написать тест, который проверяет поведение системы при дублировании запросов [1].

#### ✅ Best Practices (Стандарты 2026)
1.  **Contract Testing для микросервисов:** Использование инструментов вроде **Pact** или средств для линтинга Protobuf-файлов. До написания E2E-логики SDET проверяет, что потребитель (Consumer) и поставщик (Provider) API "говорят на одном языке" без поднятия реальных серверов [7][8].
2.  **Edge-Case Testing (Проверка Timeouts):** При тестировании идемпотентных платежных POST-запросов, Senior SDET симулирует сброс сети (Connection Drop) *после* того, как запрос ушел на сервер, но *до* получения ответа. Затем автотест отправляет запрос повторно с тем же `Idempotency-Key` и проверяет, что в БД создана строго одна транзакция [1][4].
3.  **Гибридный подход (Hybrid Architecture):** В 2026 году нормально иметь смешанные системы. SDET использует REST (Playwright/requests) для тестирования внешних клиентских API и gRPC-клиенты для валидации внутреннего состояния микросервисов в рамках одного тестового сценария[3][5].
