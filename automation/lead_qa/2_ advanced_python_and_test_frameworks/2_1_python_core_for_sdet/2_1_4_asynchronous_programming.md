## ⚙️ 1. Как работает Event Loop (Цикл событий) под капотом

Главное заблуждение: *"Асинхронность — это выполнение задач параллельно в нескольких потоках"*. 
**Нет. В классическом `asyncio` всё работает строго в ОДНОМ потоке (Single Thread).**

Как один поток может выполнять тысячи запросов "одновременно"? За счет **Кооперативной многозадачности (Non-blocking I/O)**.

**Как это работает (Under the Hood):**
1. В центре `asyncio` крутится **Event Loop (Цикл событий)** — грубо говоря, это бесконечный `while True` цикл.
2. Под капотом Event Loop использует низкоуровневые возможности операционной системы (системные вызовы `epoll` в Linux или `kqueue` в macOS/BSD).
3. Когда твоя функция делает сетевой запрос (I/O операция, например, ждет ответа от API), она доходит до слова `await`.
4. `await` — это сигнал: *"Я жду данные по сети, процессор мне пока не нужен. Забирай управление!"*.
5. Event Loop ставит эту функцию на паузу (сохраняя её состояние) и переключается на другую задачу (которая, например, уже дождалась ответа и готова парсить JSON).

> 🔥 **Архитектурный инсайт:** Асинхронность работает только для **I/O-bound** задач (сеть, файловая система, база данных). Если ты запустишь в Event Loop **CPU-bound** задачу (например, шифрование RSA, генерацию крипто-хэшей или блокирующий `time.sleep(10)`), ты "заморозишь" весь Event Loop! Ни один другой тест не выполнится, пока процессор не закончит считать.

## 🧠 2. Корутины, Таски и Фьючерсы (Словарь Архитектора)

На собеседованиях часто просят объяснить разницу между этими тремя понятиями:

1. **Coroutine (Корутина):** Функция, объявленная через `async def`. Сама по себе она ничего не делает. Если написать `my_coro()`, она не выполнится, а просто вернет объект корутины. Чтобы запустить её, нужен `await`.
2. **Task (Задача):** Это обертка над корутиной (создается через `asyncio.create_task()`). Как только ты обернул корутину в Task, она немедленно планируется (сшедулится) в Event Loop для выполнения на фоне.
3. **Future (Фьючерс):** Низкоуровневый объект. Это "обещание" того, что результат появится в будущем (аналог Promise в JavaScript). Task — это подкласс Future.

## 🚀 3. Тренды 2025-2026: Управление конкурентностью (`TaskGroup` vs `gather`)

Представь задачу: тебе нужно в автотесте создать 100 тестовых пользователей через API. Делать это циклом `for` через `await` (последовательно) — долго. Нам нужна **конкурентность**.

**Старый подход (до Python 3.11):** Использование `asyncio.gather()`. Его проблема в том, что если одна из 100 задач падает с ошибкой, остальные продолжают выполняться "сиротами", что может привести к утечкам ресурсов (например, открытым соединениям к БД).

**Modern Way (Python 3.11+ / 2025 год):** Использование **`asyncio.TaskGroup`**. Это асинхронный контекстный менеджер, который гарантирует "Структурную конкурентность". Если одна задача падает, он корректно отменяет (cancel) все остальные, предотвращая утечки.

### 💻 Пример: Создание 100 пользователей конкурентно (уровень Senior)

```python
import asyncio
import time
import httpx

async def create_user(client: httpx.AsyncClient, user_id: int) -> int:
    """Асинхронный HTTP запрос"""
    # Эмуляция: ждем ответ от API 0.5 секунды
    await asyncio.sleep(0.5) 
    # В реальности тут: response = await client.post(...)
    print(f"Пользователь {user_id} создан")
    return user_id

async def main_test_setup():
    start_time = time.time()
    
    # 1. Открываем клиент один раз для всех запросов (экономия на Handshake)
    async with httpx.AsyncClient() as client:
        
        # 2. Используем TaskGroup для безопасного конкурентного запуска (Python 3.11+)
        async with asyncio.TaskGroup() as tg:
            tasks =[]
            for i in range(1, 101):
                # Добавляем задачи в группу. Они начинают выполняться СРАЗУ на фоне!
                task = tg.create_task(create_user(client, i))
                tasks.append(task)
                
        # 3. Как только блок 'async with' завершится, мы ГАРАНТИРОВАННО 
        # дождемся выполнения всех задач.
        
        results = [task.result() for task in tasks]
        
    end_time = time.time()
    print(f"\n[INFO] Создано {len(results)} юзеров за {end_time - start_time:.2f} сек.")
    # Итог: 100 запросов по 0.5 сек выполнятся суммарно за ~0.55 сек, а не за 50 секунд!

# Запуск Event Loop
# asyncio.run(main_test_setup())
```

## ⚡ 4. Ускорение (Hack для Лидов): `uvloop`

Если на собеседовании речь зайдет про нагрузочное тестирование на Python (например, написание кастомного раннера для API или использование Locust с Async-плагинами), обязательно упомяни **`uvloop`**.

Встроенный в Python Event Loop написан на чистом Python и работает стабильно, но не идеально быстро. `uvloop` — это drop-in замена (полный аналог), написанная на Cython поверх библиотеки `libuv` (той самой, на которой работает Node.js).
Внедрение `uvloop.install()` в `conftest.py` делает асинхронные тесты быстрее в 2-4 раза (по части обработки I/O событий).

## 🎤 Как это спросят на собеседовании

> **Вопрос 1: "В твоем асинхронном автотесте есть такой код: `time.sleep(5)`. Что произойдет с тестовым прогоном, если ты запустишь тесты параллельно?"**
* **Ответ Лида:** "`time.sleep()` — это синхронная (блокирующая) функция. Поскольку `asyncio` работает в одном потоке, вызов `time.sleep()` полностью заблокирует Event Loop. Все остальные асинхронные тесты, которые в этот момент ждали ответа по сети, 'замрут' на 5 секунд. Это грубейшая архитектурная ошибка. Для ожиданий в асинхронном коде я обязан использовать только неблокирующий `await asyncio.sleep(5)`."

> **Вопрос 2: "Чем `asyncio.TaskGroup` отличается от `asyncio.gather`?"**
* **Ответ Лида:** "Концепцией **Структурной конкурентности**. Если я передам 10 запросов в `asyncio.gather()`, и третий запрос упадет с исключением, `gather` выбросит ошибку, но остальные 9 запросов продолжат выполняться в фоне ('осиротеют'). Это может сломать консистентность тестовых данных. `TaskGroup` работает как контекстный менеджер: если падает хотя бы одна задача, он автоматически рассылает сигнал отмены (`CancelledError`) всем остальным задачам в группе, безопасно закрывая ресурсы. Это стандарт начиная с Python 3.11."

> **Вопрос 3: "Мне нужно запустить в рамках асинхронного теста CPU-bound задачу. Например, сгенерировать 10 000 криптографических токенов для нагрузки. Если это 'заморозит' Event Loop, как это архитектурно обойти?"**
* **Ответ Лида:** "Чтобы тяжелая математика не блокировала Event Loop, её нужно вынести из основного потока. В `asyncio` для этого есть метод `loop.run_in_executor()`. Я создам пул процессов (например, `ProcessPoolExecutor`) и передам эту синхронную CPU-bound функцию в него. Event Loop будет просто асинхронно ждать (`await`), пока соседний процесс не закончит вычисления, и при этом продолжит обрабатывать сетевые API-вызовы тестов."
